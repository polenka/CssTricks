1. HTML5 и основные теги (header, footer, article, aside, section, nav). Новые типы и возможности инпутов (email, url, placeholder). Не забываем про автоматическую валидацию специальных полей, она может портить дизайн.
Source: Марк Пилгрим Погружение в HTML5

2. - Каков рецепт кроссбраузерной верстки? Верстайте сразу для всех поддерживаемых браузеров. Только для всех и сразу. Поддержка браузеров в той или иной степени. Либо использование принципа деградации для устаревших версий. Правила на основе Modernizr (поможет не только для ИЕ, но и с определением тач-скрина, включенности/отключенности js и т.п.) - HTML5 для ИЕ: 

<!--[if lt IE 9]>
<script src="http://html5shim.googlecode.com/svn/trunk/html5.js">lt;/script>
<![endif]-->
- IE conditional classes: 
<!--[if lt IE 7 ]> <html class="ie6"> <![endif]-->
<!--[if IE 7 ]><html class="ie7"> <![endif]-->
<!--[if IE 8 ]><html class="ie8"> <![endif]-->
<!--[if IE 9 ]><html class="ie9"> <![endif]-->
<!--[if (gt IE 9)|!(IE)]><!--><htmlclass=""><!--<![endif]-->
ИЛИ привычные IE conditional stylesheets. Или хотя бы сообщение о неподдержке.

- PIE - http://css3pie.com При необходимости.

10. Браузерные (вендорные) префиксы: -moz, -webkit, -ms, -o. Свойство БЕЗ префикса - всегда ПОСЛЕДНЕЕ. О валидности css пока речь не идет.

11. Минимум пустых элементов. Пустые, желательно inline теги, используют для иконочных шрифтов, спрайтов (желательно какой-то текст все же использовать), для элементов дизайна, которые ну никак не разместить для имеющихмя тегов. :) Скажем НЕТ! очищению потока с помощью пустых дивов.

12. Использование clearfix. 

.clearfix:after {
	content: " ";
	display: block;
	height: 0;
	overflow: hidden;
	visibility: hidden;
	width: 0;
}
.clearfix {zoom: 1;}
.clearfix:after {
	visibility: hidden;
	display: block;
	font-size: 0;
	content: " ";
	clear: both;
	height: 0;
}
.clearfix {zoom: 1;}

16. Валидация. HTML не должен содержать ошибок/предупреждений. В качестве исключений можно оставить обоснованные ошибки (например, target=_blank);

в CSS обращаем внимание только на синтаксические ошибки. Вопрос с невалидными свойствами для ИЕ открыт. Если используете условные классы, то не смысла перетаскивать в другой файл.

17. В некоторых случаях убодно выносить отдельные стили в отдельные классы и затем собирать внешний вид элементы из этих классов, как конструктор. Чаще выносят декоративные стили, но иногда и позиционные.

18. !important - зло. Но иногда необходимое (для переопределения inline css). inline css без js - тоже зло!

19. Аккуратность - отстутствие "схлопнутых дивов" (если это не специально сделано), разных отступов у однотипных элементов.

20. О чем не всегда думают программисты: контент без тегов, теги для отличных элементов без классов (очень часто у сообщений об ошибках), изменение чего-либо js, используя только css (пример: открылся, закрылся элемент по нажатию на кнопку. Только показать и скрыть его в inline css чаще всего мало. Обычно и внешний вид кнопки надо менять и что-нибудь еще. Поэтому мне удобнее менять-добавлять классы hidden/visible у элемента и добавлять еще какие-нибудь классы у родителя, в идеале у родителя, общего для появляющегося элемента и, например, того, по действию на который, он появляется).
